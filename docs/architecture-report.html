01

개요

사용자가 자연어로 매출 관련 질문을 하면, Gemini가 MCP(Model Context Protocol) 서버가 노출하는 도구만을 사용해 MS-SQL의 화이트리스트 뷰를 조회하고, 집계·필터·기간 비교 결과를 대화와 차트로 답변한다. 에이전트는 DB 커넥션 문자열을 직접 다루지 않으며, 원본 테이블에도 접근할 수 없다 — 접근 가능한 대상은 사전 정의된 매출 뷰(View)로 한정된다.

설계 의도

대화 엔진(Gemini)과 DB 자격증명을 프로세스 경계로 분리해, 에이전트가 잘못되거나 악의적인 SQL을 생성하더라도 실행 전 다단계 가드레일을 통과해야만 실행되도록 한다.

기본값

새 뷰·새 컬럼 노출은 모두 명시적 화이트리스트 등록이 필요하다. 등록되지 않은 대상은 기본적으로 조회 불가 — "허용 목록" 모델이지 "차단 목록" 모델이 아니다.

02

요청 흐름

브라우저의 챗 메시지 한 건이 답변으로 돌아오기까지 거치는 경로다.

브라우저
Chat.tsx — 사용자 메시지 입력
Next.js rewrite
frontend/next.config.js — /api/* → BACKEND_URL/api/* (기본 http://localhost:8000, proxyTimeout 120s)
FastAPI POST /api/chat
backend/app/main.py
run_chat()
backend/app/gemini_agent.py — 대화 1턴마다 새 MCP 세션 open
McpSalesClient (stdio)
backend/app/mcp_client.py → python -m backend.mcp_server.server 서브프로세스
MCP 서버 → MSSQL
backend/mcp_server/server.py — 화이트리스트 뷰만 SELECT 실행 후 세션 close
⚠

MCP 세션은 요청마다 새로 열고 닫는다. 앱 전체에서 세션을 공유하면 동시 채팅 요청이 같은 stdio 파이프를 두고 경합해 서로의 도구 호출을 직렬화시킨다. 이는 의도된 설계이며, 공유/풀링 세션으로 "최적화"하는 변경은 지양한다.

03

Gemini 도구 호출 루프

gemini_agent.py는 매 요청마다 MCP 서버의 도구 목록을 Gemini FunctionDeclaration으로 변환한 뒤, 다음 사이클을 최대 MAX_TOOL_ROUNDS = 6회 반복한다. Gemini가 function_call 없이 텍스트만 반환하면 루프를 종료하고 그 텍스트를 답변으로 반환한다.

1
generate_content 호출

시스템 프롬프트(SYSTEM_PROMPT, config.py)와 대화 히스토리를 Gemini에 전달. Gemini는 SQL 작성 전 get_view_aliases / get_column_aliases로 한글 별칭을 실제 뷰·컬럼명으로 해석하도록 지시받는다.

2
function_call 실행

응답에 담긴 함수 호출을 MCP 세션에 위임 — mcp_client.call_tool()이 실제 도구를 실행하고 결과를 텍스트로 회수한다.

3
function_response 피드백

도구 실행 결과를 function_response 파트로 대화 컨텍스트에 추가하고 다시 generate_content를 호출한다.

또한 시스템 프롬프트는 추이·비교 분석 시 프론트엔드가 파싱할 수 있도록 ```chart``` JSON 블록을 함께 출력하도록 Gemini에 지시한다.

04

접근 통제 체인

접근 통제는 단일 검사가 아니라 3단계 defense-in-depth다. 각 단계는 앞 단계가 제공하는 보장을 전제로 동작하므로, 어느 한 단계의 정규식이나 로직을 느슨하게 바꾸면 나머지가 조용히 깨질 수 있다.

1
화이트리스트 & 별칭 치환 — views_whitelist.py

SALES_VIEW_WHITELIST가 조회 가능한 schema.view의 단일 소스(source of truth)다. 새 매출 뷰는 이 파일에만 추가하면 된다. 같은 파일이 한글 별칭 테이블(VIEW_ALIASES, COLUMN_ALIASES)과 rewrite_query_with_aliases()를 보유하며, SQL 문자열 내 한글 별칭을 sql_guard 검증 이전에 실제 이름으로 치환한다.

2
SQL 가드레일 — sql_guard.py

validate_and_prepare()가 별칭 치환 이후 순서대로 실행: 다중 문장(;) 차단 → SELECT 이외 차단 → 금지 키워드(DML/DDL, xp_cmdshell, sp_executesql, OPENROWSET 등) 차단 → 모든 FROM/JOIN 대상을 화이트리스트와 대조 → TOP 절 없으면 TOP 1000 자동 삽입.

3
DB 계정 권한 — db.py

MSSQL_READONLY_USER가 화이트리스트 뷰에 대해서만 SELECT 권한을 갖는 DB 계정이라는 운영상 전제조건이다. 애플리케이션 코드가 이를 강제하지는 않으므로, DB 계정 권한 설정 자체가 마지막 방어선이다.

⚠

가드를 수정할 때는 별칭 치환 → 키워드 검사 → 테이블 참조 검사 → 행수 제한 전체 파이프라인 관점에서 우회 가능성을 검토해야 한다. 정규식 하나만 보고 안전성을 판단하지 말 것.

05

노출 MCP 도구

MCP 서버(backend/mcp_server/server.py)는 정확히 5개 도구만 노출한다. DB 자격증명은 이 서버 프로세스만 알고 있다.

도구역할
list_sales_views접근 가능한 매출 뷰 목록 반환
get_view_schema지정 뷰의 컬럼명·타입 스키마 반환 (화이트리스트 외 뷰는 거부)
get_view_aliases한글 뷰 별칭 ↔ 실제 뷰 이름 매핑 반환
get_column_aliases지정 뷰의 한글 컬럼 별칭 반환
run_sql화이트리스트 뷰에 대한 SELECT 전용 쿼리 실행 (읽기 전용)

서브프로세스 실행 시 mcp_client.py가 부모 프로세스의 전체 환경변수를 명시적으로 전달한다 — MCP의 기본 서브프로세스 환경은 OS 화이트리스트(PATH, APPDATA 등)만 상속하므로, 이를 생략하면 MSSQL_* / GEMINI_* 값이 서브프로세스에 전달되지 않는다.

06

보안 설계 원칙

최소 권한

DB 접속 계정은 해당 매출 뷰에 대한 SELECT 권한만 보유. 인사·급여·개인정보 등 매출 외 데이터가 포함된 테이블/뷰는 조회 대상에서 원천 배제.

인젝션 방지

파라미터화 쿼리를 사용하며, 사용자 입력을 쿼리 문자열에 직접 결합하지 않는다. 다중 문장·DDL/DML·시스템 프로시저는 sql_guard에서 사전 차단.

자격증명 격리

MCP 서버 설정(접속 정보, 자격증명)은 저장소에 커밋하지 않고 환경변수로 관리. Gemini 에이전트 프로세스는 DB 자격증명 자체를 보유하지 않는다.

기본값 거부

새 매출 뷰 추가는 화이트리스트에 명시적으로 등록해야 하며, 등록되지 않은 대상의 기본 상태는 언제나 "접근 불가".

07

프론트엔드

기본형은 대화형 챗봇 UI다. frontend/components/Chat.tsx가 react-markdown으로 assistant 메시지를 렌더링하며, language-chart 태그가 붙은 코드 블록을 가로채 ChartRenderer.tsx의 parseChartSpec으로 전달해 표·차트 형태의 시각화를 함께 보여준다.
